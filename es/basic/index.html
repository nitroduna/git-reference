book Snapshotting Básico

En Git todo consiste en crear y guardar snapshots de nuestro proyecto, para despúes trabajar con estos snapshots o compararlos unos con otros. Este apartado decribe los comandos necesarios para construir y grabar (commit) los snapshots de nuestro proyecto.

Un concepto importante del snapshotting es que Git mantiene un 'índice', que actúa como una especie de área de montaje (staging area) de nuestro snapshot. Esto nos permite acumular una serie de instantaneas bien construidas desde los ficheros modificados en nuestro directorio de trabajo (working directory), en lugar de tener que realizar de golpe el commit de todos los cambios de los ficheros.

En resumidas cuentas, usaremos git add para comenzar con el seguimiento (tracking) de nuevos ficheros, pero también para añadir (stage) al área de montaje cambios nuevos en ficheros que ya se habían añadido, después usaremos git status y git diff para examinar lo que ha sido modificado y organizado en el área de montaje, y finalmente usaremos git commit para realizar la grabación de nuestro snapshot en el historial. Este será el flujo de trabajo básico que estaremos utilizando la mayor parte del tiempo.

docs   book git add añade el contenido de los ficheros al área de montaje

En Git tenemos que añadir los ficheros a nuestro staging area (área de montaje) antes de que se pueda realizar su commit (grabarlos en un snapshot). Si se trata de un fichero nuevo podemos ejecutar git add para añadirlo por primera vez al staging area, pero incluso cuando el fichero ya lo tenemos "bajo seguimiento" (tracked) - si por ejemplo estaba en nuestro último commit - será igualmente necesario ejecutar git add para añadir las nuevas modificaciones del fichero al staging area. Veamos algunos ejemplos de esto.

Volviendo a nuestro ejemplo Hello World, una vez que hemos inicializado el projecto empezaríamos a añadir nuestros ficheros y lo haríamos con git add. Podemos usar git status para ver en que estado está nuestro proyecto. 

$ git status -s
?? README
?? hello.rb
En este momento tenemos dos ficheros que no están bajo seguimiento (untracked files). Pero podemos añadirlos al staging area. 
$ git add README hello.rb
Si ahora ejecutamos de nuevo git status, veremos como han sido añadidos. 
$ git status -s
A  README
A  hello.rb

También es habitual añadir recursivamente todos los ficheros de un nuevo proyecto, especificando el directorio de trabajo actual como por ejemplo con git add .. Es decir, Git añadirá recursivamente todos los ficheros que estén dentro del directorio entregado como parametro en el git add, si le entregamos el directorio de trabajo actual, Git sencillamente empezará el seguimiento (tracking) de todos los ficheros en este directorio. En este ejemplo, un git add . estaría haciendo lo mismo que un git add README hello.rb, y en este caso en particular también podríamos hacer git add *, pero sólo porque no tenemos subdirectorios en los que el * pueda entrar y añadir de forma recursiva.

Pues bién, si ahora editamos uno de estos ficheros y ejecutamos git status de nuevo, nos encontraremos algo extraño. 

$ vim README
$ git status -s
AM README
A  hello.rb

El estado 'AM' significa que los ficheros han sido modificados en el disco desde la última vez que fueron añadidos (al staging area). Lo que significa que si ahora mismo hicieramos un commit para crear nuestro snapshot, estaríamos grabando la versión del fichero que teníamos la última vez que ejecutamos el git add, y no la versión que tenemos en nuestro disco. Git no asume que el fichero tal y como está en el disco es lo que queremos añadir a nuestro snapshot - tenemos que decirselo con el comando git add (aunque ya estubiera añadido).

En resumidas cuentas, ejecutamos git add en un fichero cuando queremos incluir en el siguiente snapshot - que guardemos con el commit - cualquier cambio que hayamos realizado en dicho fichero. Cualquier cosa que cambiemos en un fichero y que no se haya añadido con un git add no será incluída en el snapshot - lo que significa que podemos elaborar nuestros snapshots con algo de más precisión que con la mayoría de los sistemas SCM.

Para un ejemplo bastante interesante de como usar esta flexibilidad de añadir al area de montaje (stage) sólo partes de ficheros modificados, se puede consultar la descripción de la opción '-p' de git add en el libro Pro Git.

docs   book git status muestra el estado de los ficheros en el directorio de trabajo y área de montaje

Como vimos en el apartado git add, para ver el estado de nuestro staging area (area de montaje) comparado con el código en nuestro directorio de trabajo (working directory), podemos ejecutar el comnado git status. Lo hemos visto con la opción -s, que nos proporciona una salida en en formato corto. Sin este flag, el comando git status nos mostrará más informacíón y algunas pequeñas ayudas. Veamos las diferencias entre del informe de estado pero sin la opción -s. La salida de formato corto tiene la siguiente apariencia: 

$ git status -s
AM README
A  hello.rb
Mientras que la misma información en formato largo tiene la siguiente apariencia: 
$ git status
# On branch master
#
# Initial commit
#
# Changes to be committed:
#   (use "git rm --cached ..." to unstage)
#
# new file:   README
# new file:   hello.rb
#
# Changed but not updated:
#   (use "git add ..." to update what will be committed)
#   (use "git checkout -- ..." to discard changes in working directory)
#
# modified:   README
#

Podemos observar que la salida en formato corto es bastante más compacta, en cambio la salida en formato largo ofrece consejos y pistas útiles que podríamos necesitar a continuación.

Git también nos informará sobre ficheros que fueron borrados desde el último commit o ficheros modificados o añadidos al staging area desde nuestro último commit.

$ git status -s
M  README
 D hello.rb
Podemos ver que en el formato corto hay dos columnas. La primera columna es para el staging area, la segunda es para el working directory. A si que si por ejemplo tenemos el fichero README en el staging area y lo modificamos de nuevo sin ejectuar git add una segunda vez, veremos lo siguiente: 
$ git status -s
MM README
 D hello.rb

En resumidas cuentas, ejecutamos git status para ver si algo ha sido modificado y/o añadido al staging area desde el último commit, para que podamos decidir si queremos hacer el commit y crear un nuevo snapshot y también decidir lo grabaremos en dicho snapshot.

docs   book git diff muestra la diferencia entre lo que está en el staging area y lo que se ha modificado sin añadir al staging area

Existen dos formas principales de usar el comando git diff. Una de ellas será descrita en este apartado, pero la otra se explicará más adelante en el apartado "Inspection and Comparison". El sistema que vamos a usar en este apartado sirve para mostrar cambios entre lo que hemos añadido al staging area y lo que se ha modificado en el disco pero no se ha añadido a dicho staging area.

git diff muestra las diferencias de los cambios no añadidos al staging area (unstaged changes)

Sin ningún argumento adiccional, un sencillo git diff mostrará en el formato diff unificado (un parche) qué código o contenido ha sido cambiado en nuestro proyecto desde el último commit y que aun no ha sido añadido al staging area para el commit que guardará el siguiente snapshot. 

$ vim hello.rb
$ git status -s
 M hello.rb
$ git diff
diff --git a/hello.rb b/hello.rb
index d62ac43..8d15d50 100644
--- a/hello.rb
+++ b/hello.rb
@@ -1,7 +1,7 @@
 class HelloWorld
   
   def self.hello
-    puts "hello world"
+    puts "hola mundo"
   end
 
 end

Mie entras que git status mostrará los ficheros que han sido cambiados y/o añadidos al staging area desde nuestro último commit, git diff mostrará cuales son realmente esos cambios, linea a linea. Por lo general se trata de un buen "comando que siempre ejecutamos despues" de un git status.

git diff --cached muestra las diferencias de los cambios añadidos al staging area (staged changes)

El comando git diff --cached mostrará los contenidos que han sido añadidos al staging area (staged). Es decir, mostrará lo cambios que irán en el snapshot guardado con el siguiente commit. De esta forma, si fueramos a añadir al staging area los cambios del fichero hello.rb del ejemplo de arriba, git diff por si mismo no mostrará ninguna salida, ya que sólo mostrará lo que aun no ha sido añadido al staging area. 

$ git status -s
 M hello.rb
$ git add hello.rb 
$ git status -s
M  hello.rb
$ git diff
$

Para ver los cambios añadidos al área de montaje, podemos ejecutar git diff --cached.

$ git status -s
M  hello.rb
$ git diff
$ 
$ git diff --cached
diff --git a/hello.rb b/hello.rb
index d62ac43..8d15d50 100644
--- a/hello.rb
+++ b/hello.rb
@@ -1,7 +1,7 @@
 class HelloWorld
   
   def self.hello
-    puts "hello world"
+    puts "hola mundo"
   end
 
 end

git diff HEAD muestra las diferencias de todos los cambios añadidos o no añadidos al area de montaje (staged or unstaged) changes

Si queremos ver ambios cambios juntos, los añadidios o los no añadidos al area de montaje, podemos ejecutar diff HEAD - básicamente signigica que queremos ver las diferencias entre nuestro directorio de trabajo y el último commit, ignorando el staging area. Si hacemos cualquier otro cambio a nuestro fichero hello.rb tendremos algunos cambios en el área de montaje, y otros cambios fuera del area de montaje. Esto es lo que los tres comandos diff mostraran:

$ vim hello.rb 
$ git diff
diff --git a/hello.rb b/hello.rb
index 4f40006..2ae9ba4 100644
--- a/hello.rb
+++ b/hello.rb
@@ -1,7 +1,7 @@
 class HelloWorld
   
+  # says hello
   def self.hello
     puts "hola mundo"
   end
 
 end
$ git diff --cached
diff --git a/hello.rb b/hello.rb
index 2aabb6e..4f40006 100644
--- a/hello.rb
+++ b/hello.rb
@@ -1,7 +1,7 @@
 class HelloWorld
 
   def self.hello
-    puts "hello world"
+    puts "hola mundo"
   end
 
 end
$ git diff HEAD
diff --git a/hello.rb b/hello.rb
index 2aabb6e..2ae9ba4 100644
--- a/hello.rb
+++ b/hello.rb
@@ -1,7 +1,8 @@
 class HelloWorld
 
+  # says hello
   def self.hello
-    puts "hello world"
+    puts "hola mundo"
   end
 
 end

git diff --stat muestra un resumen de los cambios en lugar de un listado completo de diferencias

Si no queremos un listado complet ode diferencias, pero queremos más de los que nos ofrece el comando git status, podemos usar la opción --stat, la cual nos muestra un resumen en su lugar. Aqui tenemos el mismo ejemplo de arriba, pero usando la opción --stat option. 

$ git status -s
MM hello.rb
$ git diff --stat
 hello.rb |    1 +
 1 files changed, 1 insertions(+), 0 deletions(-)
$ git diff --cached --stat
 hello.rb |    2 +-
 1 files changed, 1 insertions(+), 1 deletions(-)
$ git diff HEAD --stat
 hello.rb |    3 ++-
 1 files changed, 2 insertions(+), 1 deletions(-)

También podemos proporcionar la ruta de un fichero al final de cualquiera de estas opciones, para limitar la salida del comando diff a un fichero o subdirectorio específico.

En resumidas cuentas, ejecutamos git diff para mostrar en detalle la salida del comando git status - cómo han sido modificados o añadidos al área de montaje linea por línea.

docs   book git commit graba un snapshot del staging area

Una vez que hemos añadido al área de montaje con el comando git add el contenido que queremos incluír en el siguiente snapshot, podemos ejecutar git commit para realmente guardar the snapshot. Git graba nuestro nombre y dirección de email con cada commit que hacemos, así que el primer paso es decirle a Git cuales son estos. 

$ git config --global user.name 'Your Name'
$ git config --global user.email you@somedomain.com

Vamos a añadir al área de montaje y luego el commit de todos los cambios de nuestro fichero hello.rb. En este primer ejemplo usaremos la opción-m para proporcionar el mensaje del commit en la linea de comando. 

$ git add hello.rb 
$ git status -s
M  hello.rb
$ git commit -m 'my hola mundo changes'
[master 68aa034] my hola mundo changes
 1 files changed, 2 insertions(+), 1 deletions(-)

Ahora es cuando hemos grabado el snapshot. Si ejecutamos git status de nuevo, veremos que ahora tenemos un "directorio de trabajo limpio", lo que significa que no hemos realizado ningún cambio desde nuestro último commit - no tenemos ningún trabajo pendiente por incluir en algún snapshot en nuestro checkout.

$ git status
# On branch master
nothing to commit (working directory clean)

Si no ponemos la opción -m, Git tratará de abrir un editor de texto para que escribamos el mensaje del commit. En vim, que es el editor que se utilizará por defecto si no lo hemos modificado en la configuración, la pantalla podría tener la siguiente apariencia: 


# Please enter the commit message for your changes. Lines starting
# with '#' will be ignored, and an empty message aborts the commit.
# On branch master
# Changes to be committed:
#   (use "git reset HEAD ..." to unstage)
#
# modified:   hello.rb
#
~
~
".git/COMMIT_EDITMSG" 9L, 257C

En este punto podemos añadir nuestro mensaje de commit al principio del documento. Cualquier linea que empiece por '#' será ignorada - Git mostrará el mensaje en la salida del comando git status como un recordatorio de lo que hemos modificado y añadido al área de montaje.

En general, es muy importante escribir buenos mensajes de commit. Para proyectos open source, normalmente es una regla escribir nuestro mensaje mas o menos con el siguiente formato:

Short (50 chars or less) summary of changes

More detailed explanatory text, if necessary.  Wrap it to about 72
characters or so.  In some contexts, the first line is treated as the
subject of an email and the rest of the text as the body.  The blank
line separating the summary from the body is critical (unless you omit
the body entirely); some git tools can get confused if you run the
two together.

Further paragraphs come after blank lines.

 - Bullet points are okay, too

 - Typically a hyphen or asterisk is used for the bullet, preceded by a
   single space, with blank lines in between, but conventions vary
   here

# Please enter the commit message for your changes. Lines starting
# with '#' will be ignored, and an empty message aborts the commit.
# On branch master
# Changes to be committed:
#   (use "git reset HEAD ..." to unstage)
#
# modified:   hello.rb
#
~
~
~
".git/COMMIT_EDITMSG" 25L, 884C written

El mensaje del commit es muy importante. Ya que una buena parte de la potencia de Git es su flexibilidad en fabricar cuidadosamente los commits de forma local para despues compartirlos, es bastante potente ser capaz de escribir tres o cuatro commits de cambios de lógica separados de forma que nuestro trabajo pueder más fácilmente revisado por otros. Ya que hay una separación entre el commit y el push de esos cambios, deberíamos tomarnos el tiempo necesario para hacer más fácil a la gente con la que trabajamos mostrar lo que hemos hecho, situando cada cambio de lógica en commits separados con un buen mensaje de commit de forma que sea más facil para ellos ver lo que estamos haciendo y porqué lo estamos haciendo.

git commit -a añade automáticamente al área de montaje todos los ficheros que están en seguimiento, ficheros modificados antes de el commit

Si el paso de añadir las modificaciones al área de montaje con el git add stage no parece demasiado molesto, Git nos permite saltar esta parte del flujo de trabajo con la opción -a. Básicamente le dice a Git que ejecute git add en cualquiera de los ficheros que están bajo "seguimiento" (tracked files) - es decir, cualquier fichero que esttubiera en nuestro último commit y que haya sido modificado. Esto nos permite realizar un estilo de flujo de trabajo más parecido al de Subversion, simplemente editandos los ficheros y luego ejecutando git commit -a cuando queremos crear un snapshot de todo lo que ha cambiado. Aun es necesario ejecutar git add para empezar el seguimiento de nuevos ficheros, aunque esto sería igual que en Subversion. 

$ vim hello.rb
$ git status -s
 M  hello.rb
$ git commit -m 'changes to hello file'
# On branch master
# Changed but not updated:
#   (use "git add ..." to update what will be committed)
#   (use "git checkout -- ..." to discard changes in working directory)
#
# modified:   hello.rb
#
no changes added to commit (use "git add" and/or "git commit -a")
$ git commit -am 'changes to hello file'
[master 78b2670] changes to hello file
 1 files changed, 2 insertions(+), 1 deletions(-)

Daros cuenta de que si no añadimos al area de montaje los cambios y después ejecutamos git commit, Git simplemente nos mostrará la salida del comando git status , recordandonos que no hay nada pendiente en el área de montaje. He resaltado la parte importante de ese mensaje, que dice que no se ha añadido nada que pueda grabarse con un commit. Si usamos la opción -a , añadirá y realizará el commit de todos los cambios de golpe.

Con esto ya tenemos completo el flujo de trabajo del snapshotting - hacemos cambios a nuestros ficheros, después usamos git add para añadir al staging area los ficheros que queremos modificar, git status y git diff para ver lo que hemos cambiado, y finalmente git commit para realmente guardar el snapshot para siempre.

En resumidas cuentas, ejecutamos git commit para grabar el snapshot del contenido que hemos añadido al área de montaje. Este snapshot puede ser entonces comparado, compartido y deshecho en caso de que lo necesitemos.

docs   book git reset HEAD sacar del area de montaje los cambios que fueron añadidos

git reset es probablemente el comando más confuso alguna vez escrito por humanos. He usado Git durante años, incluso escrito un libro sobre Git y aun a veces me sigue liando. Por lo tanto simplemente vamos a ver las tres invocaciones específicas de éste que suelen ser las más utiles y os recomendaría siemplemente usarlas a ciegas tal y como yo hago - ya que puede ser util.

En este caso, pdemos usarlo para eliminar del área de montaje algo que accidentalmente hayamos añadido. Supongamos que hemos modificado dos ficheros y queremos grabarlos en dos commits diferentes. Deberíamos añadirlos al área de montaje y hacer el commit de uno de ellos, después, repetir lo mismo con el otro. Si accidentalmente añadimos al staging area los dos, ¿como podemos sacar del area de montaje uno de ellos? Lo hacemos con git reset HEAD -- fichero. Técnicamente no es necesario añadir el -- - se usa para decirle a Git que termina el listado de opciones y empieza el listado de rutas de ficheros, aunque es bueno habiturase a usarlo para separar las opciones de los ficheros incluso aunque no sea necesario.

Bién, veamos los pasos sacando algo del área de montaje. Aquí tenemos dos ficheros que han sido modificados desde nuestro último commit. Añadiremos ambos al área de montaje y a continuación sacaremos uno de ellos.

$ git status -s
 M README
 M hello.rb
$ git add .
$ git status -s
M  README
M  hello.rb
$ git reset HEAD -- hello.rb 
Unstaged changes after reset:
M hello.rb
$ git status -s
M  README
 M hello.rb

Si ahora ejecutamos git commit sólo se grabarán los cambios del fichero README, y no los hello.rb que ya no está en el área de montaje.

Para satisfacer vuestra curiosidad, lo que realmente estamos haciendo aqui es reinicializar el checksum de la entrada de ese fichero en el "índice" para que tenga lo que había en el último commit. Si git add calcula el checksums de un fichero y lo añade al "índice", git reset HEAD lo sobreescribe con lo que había antes, y por lo tanto sacandolo del staging area.

Si quisieramos simplemente poder ejecutar un git unstage, podemos configurar fácilmente un alias en Git. Tan solo ejecutamos git config --global alias.unstage "reset HEAD". Una vez hemos configurado esto, podemos ejecutar git unstage [file]en lugar del git reset.

Si olvidamos el comando para sacar algo del área de montaje, Git nos ayuda recordandonos en la salida del comando git status. Por ejemplo, si ejecutamos git status sin el -s cuando hemos añadido ficheros al área de montaje, nos informará sobre como podemos sacarlos de nuevo:

$ git status
# On branch master
# Changes to be committed:
#   (use "git reset HEAD ..." to unstage)
#
#   modified:   README
#   modified:   hello.rb
#

En resumidas cuentas, ejecutamos git reset HEAD para sacar del área de montaje ficheros sobre los que anteriormente habíamos ejecutado git add y que no queremos incluir en el commit para el siguiente snapshot.

docs   book git rm elimina ficheros del staging area

git rm eliminará entradas del staging area. Esto es un poco diferente al git reset HEAD con el cual "sacamos" ficheros del staging area (unstage). Por "sacar" me refiero a revertir el área de montaje al estado en el que estaba antes de que empezaramos a modificar cosas. Por otro lado git rm simplemente lo echa fuera del área de montaje de forma definitiva, de forma que ya no será incluido en el commit del siguiente snapshot, y por lo tanto eliminando de forma efectiva.

Por defecto, un comando git rm fichero eliminará el fichero del staging area y también de nuestro disco (del directorio de trabajo). Para dejar el fichero en el directorio de trabajo, podemos usar git rm --cached.

git mv git rm --cached orig; mv orig new; git add new

Al contrario que la mayoría de los otros sistemas de control de versiones, Git no hace el siguimiento de los cambios de nombre de ficheros. En su lugar, simplemente realiza el seguimiento de los snapshots y entonces averigua que ficheros fueron probablemente cambiados de nombre comparando los snapshots. Si un fichero fué eliminado de un snapshot y otro fichero ha sido añadido para el siguiente y su contenido es similar, Git averigua que con mucha probabilidad fue un cambio de nombre. Por tanto, aunque el comando git mv existe, es superfulo- todo lo que hace es un git rm --cached, mover el fichero en el disco, y después ejecutar un git add en el nuevo fichero. En realidad no necesitais usuarlo, pero si os resulta más facil, sentiros libre de hacerlo.

Yo personalmente no uso este comando mucho en su forma normal - para borrar ficheros. Es normalmente más fácil simplemente eliminar los ficheros del dicso y despues ejecutar un git commit -a, lo que automaticamente elimnará los ficheros de tu índice también.

En resumidas cuentas, ejecutamos git rm para eliminar ficheros del área de montaje y por tanto que dejen de ser trackeadosen Git. Tambíen los eliminará de nuestro directorio de trabajo.
